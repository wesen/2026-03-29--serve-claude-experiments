# Artifact Server Design and Implementation Guide

## Executive Summary

This document describes the design and implementation of `serve-artifacts`, a standalone Go command-line tool that serves Claude.ai artifacts — interactive web applications generated by Claude — from a local directory. The server handles two distinct artifact types: **self-contained HTML files** (served as-is) and **JSX files** (React components requiring runtime transformation). The tool is built using the Glazed command framework for consistent CLI behavior, structured output, and help system integration.

The server provides:
- A listing page showing all available artifacts with metadata
- Direct serving of HTML artifacts without transformation
- Runtime JSX-to-JavaScript transformation for React artifacts via in-browser tooling
- A hot-reload watcher for development
- The ability to add new artifacts at runtime (drop files into the directory)

## Problem Statement

Claude.ai generates two kinds of interactive artifacts:

1. **HTML artifacts**: Complete, self-contained HTML documents with inline CSS and JavaScript. These are single-file applications — everything (styles, scripts, markup) lives in one `.html` file. Examples include `QueryMac.html` (a SQL IDE) and `draft-review.html` (an article review system).

2. **JSX artifacts**: React component files that use JSX syntax and import from `react`. These are *not* directly servable by a browser — they need either a build step or a runtime transformation layer. Examples include `business-app.jsx` (a business accounting dashboard) and `editor.jsx` (a text editing tool with Mac-style windowing).

Currently, there is no convenient way to browse, serve, and interact with these artifacts locally. You would need to either open HTML files directly in a browser (losing proper URL routing) or set up a full Node.js/Vite development environment for the JSX files. We want a single Go binary that handles both types seamlessly.

### Scope

**In scope:**
- Serving HTML artifacts directly
- Serving JSX artifacts with in-browser transformation
- Listing all artifacts with metadata (name, type, size, modification date)
- Dynamic artifact discovery (new files appear without restart)
- CLI flags for port, directory, and watch mode
- Glazed command framework integration

**Out of scope (for now):**
- Server-side JSX compilation (e.g., running esbuild as a subprocess)
- Authentication or multi-user support
- Artifact editing or creation
- Deployment to production environments

## Current-State Analysis: Understanding the Artifacts

Before designing the server, you need to understand exactly what these artifacts look like and what they need to run.

### HTML Artifacts: Self-Contained Documents

Let's look at `draft-review.html` as a representative example:

```html
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
<title>DraftDesk — Article Review System</title>
<style>
  @import url('https://fonts.googleapis.com/css2?family=Chicago:wght@400&display=swap');
  :root {
    --black: #000000;
    --white: #FFFFFF;
    /* ... hundreds of lines of CSS ... */
  }
  /* All styles are inline */
</style>
</head>
<body>
  <!-- All markup is inline -->
  <div id="menubar">...</div>
  <div id="desktop">...</div>
  
  <script>
    // All JavaScript is inline — no imports, no modules
    let currentPara = null;
    function selectReaction(type) { ... }
    function submitReaction() { ... }
    // ... hundreds of lines of vanilla JS ...
  </script>
</body>
</html>
```

**Key characteristics of HTML artifacts:**

- **Completely self-contained**: All CSS, JavaScript, and HTML live in one file. There are no external script or stylesheet imports (except occasionally Google Fonts via CSS `@import`).
- **Vanilla JavaScript**: No framework, no module system, no `import` statements. Just plain DOM manipulation with `document.getElementById`, `addEventListener`, etc.
- **No build step required**: A browser can render these files as-is. You could open them with `file://` and they'd work (modulo CORS restrictions on font imports).
- **Typical size**: 500–1200 lines, roughly 20–50 KB.

**What the server needs to do for HTML artifacts:**

Essentially nothing — just serve the file with `Content-Type: text/html`. The only value the server adds is:
1. Proper HTTP URLs instead of `file://` (which matters for some browser APIs)
2. A listing/index page so you can browse available artifacts
3. Potential future features like live-reload injection

### JSX Artifacts: React Components Needing Transformation

Now let's look at `business-app.jsx`:

```jsx
import { useState, useEffect, Fragment } from "react";

// Mock data (all inline, no external API calls)
const ACCOUNTS = [
  { id: "a1", institution_name: "First National Bank", ... },
  ...
];

// Helper functions
const fmt = (n) => { ... };

// Navigation config
const NAV = [
  { id: "dashboard", label: "Dashboard", icon: "◈", group: "Overview" },
  ...
];

// Main component
export default function App() {
  const [section, setSection] = useState("dashboard");
  const [clock, setClock] = useState(new Date());

  useEffect(() => {
    const id = setInterval(() => setClock(new Date()), 30000);
    return () => clearInterval(id);
  }, []);

  return (
    <>
      <style>{CSS}</style>
      <div className="desktop">
        {/* JSX markup with event handlers, conditional rendering, etc. */}
      </div>
    </>
  );
}
```

**Key characteristics of JSX artifacts:**

- **React dependencies**: They `import` from `"react"` — specifically hooks like `useState`, `useEffect`, `useRef`, `useCallback`, and `Fragment`.
- **JSX syntax**: The `<div className="...">` syntax is not valid JavaScript. Browsers cannot parse it. It must be transformed into `React.createElement(...)` calls.
- **ES module syntax**: They use `import`/`export` statements.
- **Single-component architecture**: Each file exports one default React component (`export default function App() { ... }`). All data, styles (as template literal CSS), and sub-components are defined in the same file.
- **No external API calls**: All data is mocked inline. The components are fully self-contained in terms of data.
- **CSS-in-JS via template literals**: Styles are defined as a big string constant (e.g., `const CSS = \`...\``) and injected via `<style>{CSS}</style>`.
- **Typical size**: 600–1300 lines, roughly 25–55 KB.

**What the server needs to do for JSX artifacts:**

This is the interesting part. Browsers cannot run JSX directly. We have three options:

1. **Server-side compilation** (pre-build): Use esbuild or swc to compile JSX to plain JS before serving. This produces optimal output but requires a build tool as a dependency.
2. **In-browser transformation** (runtime): Serve the JSX source alongside a client-side transformer (like Babel standalone or Sucrase) that compiles JSX to JS in the browser before execution.
3. **Import map + ESM CDN** (hybrid): Use an import map to resolve `react` to an ESM CDN (like esm.sh or unpkg), and use a small in-browser JSX transformer.

We choose **option 3** because it requires zero build tooling, works with dynamic artifact addition, and keeps the server simple. Here is exactly how it works.

### How JSX Serving Works (The Key Technical Detail)

When a user visits `/view/business-app`, the server does NOT serve the `.jsx` file directly. Instead, it generates a **host HTML page** that:

1. Loads React and ReactDOM from an ESM CDN (esm.sh)
2. Loads Babel standalone (for in-browser JSX transformation)
3. Inlines the JSX source code inside a `<script type="text/babel">` tag
4. Babel transforms the JSX to plain JS in the browser
5. The transformed code runs and React renders the component

Here is exactly what the generated host page looks like:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>business-app — Artifact Viewer</title>
  
  <!-- Import map: tells the browser where to find "react" and "react-dom" 
       when it encounters import statements -->
  <script type="importmap">
  {
    "imports": {
      "react": "https://esm.sh/react@18.3.1",
      "react-dom": "https://esm.sh/react-dom@18.3.1",
      "react-dom/client": "https://esm.sh/react-dom@18.3.1/client",
      "react/jsx-runtime": "https://esm.sh/react@18.3.1/jsx-runtime"
    }
  }
  </script>
  
  <!-- Babel standalone: transforms JSX to JS in the browser -->
  <script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
</head>
<body>
  <div id="root"></div>
  
  <!-- The artifact's JSX source, loaded as a Babel script -->
  <script type="text/babel" data-type="module">
    // The server injects the full JSX source code here
    import { useState, useEffect, Fragment } from "react";
    
    // ... entire artifact source ...
    
    export default function App() { ... }
    
    // Auto-mount: render the default export into #root
    import { createRoot } from "react-dom/client";
    const root = createRoot(document.getElementById("root"));
    root.render(React.createElement(App));
  </script>
</body>
</html>
```

**Why this approach works well for our use case:**

- **No build tooling required**: The Go server doesn't need Node.js, esbuild, or any other tool installed. It just serves HTML and lets the browser do the work.
- **Dynamic artifact addition**: Drop a new `.jsx` file into the directory and it's immediately available. No rebuild step.
- **Faithful rendering**: The artifact runs with the same React version and behavior as on Claude.ai.
- **Trade-off**: Initial page load is slower (~500ms for Babel + React download, cached after first load). This is acceptable for a local development tool.

**Why not server-side compilation?**

Server-side compilation (embedding esbuild or calling it as a subprocess) would produce faster page loads and wouldn't require CDN access. However:
- It adds a build dependency (either a Go-native JS parser or a subprocess)
- It complicates dynamic artifact addition (need to watch + recompile)
- It's more code to write and maintain
- For a local dev tool, the 500ms cold-start penalty is negligible

We can always add server-side compilation as a future optimization.

## Detailed Component Design

### Component 1: Artifact Scanner

The scanner reads a directory and produces a list of artifact metadata.

**File:** `pkg/artifacts/scanner.go`

```go
// Artifact represents a single Claude artifact file.
type Artifact struct {
    Name         string    // filename without extension, e.g., "business-app"
    Filename     string    // full filename, e.g., "business-app.jsx"
    Type         string    // "html" or "jsx"
    Title        string    // extracted from <title> tag or component name
    Size         int64     // file size in bytes
    ModifiedAt   time.Time // last modification time
    Path         string    // absolute path on disk
}

// Scanner watches a directory for artifact files.
type Scanner struct {
    dir string
}

// NewScanner creates a scanner for the given directory.
func NewScanner(dir string) *Scanner {
    return &Scanner{dir: dir}
}

// Scan reads the directory and returns all artifacts.
// It is safe to call repeatedly (for dynamic discovery).
func (s *Scanner) Scan() ([]Artifact, error) {
    entries, err := os.ReadDir(s.dir)
    if err != nil {
        return nil, fmt.Errorf("reading artifact directory: %w", err)
    }
    
    var artifacts []Artifact
    for _, entry := range entries {
        if entry.IsDir() {
            continue
        }
        ext := filepath.Ext(entry.Name())
        var artifactType string
        switch ext {
        case ".html", ".htm":
            artifactType = "html"
        case ".jsx":
            artifactType = "jsx"
        default:
            continue // skip non-artifact files
        }
        
        info, err := entry.Info()
        if err != nil {
            continue
        }
        
        name := strings.TrimSuffix(entry.Name(), ext)
        title := extractTitle(filepath.Join(s.dir, entry.Name()), artifactType)
        
        artifacts = append(artifacts, Artifact{
            Name:       name,
            Filename:   entry.Name(),
            Type:       artifactType,
            Title:      title,
            Size:       info.Size(),
            ModifiedAt: info.ModTime(),
            Path:       filepath.Join(s.dir, entry.Name()),
        })
    }
    return artifacts, nil
}
```

**Title extraction** is type-specific:

For HTML files, we parse out the `<title>` tag:
```go
func extractHTMLTitle(path string) string {
    f, err := os.Open(path)
    if err != nil {
        return ""
    }
    defer f.Close()
    
    // Read first 4KB — title is always near the top
    buf := make([]byte, 4096)
    n, _ := f.Read(buf)
    content := string(buf[:n])
    
    // Simple regex extraction (no need for full HTML parser)
    re := regexp.MustCompile(`<title>([^<]+)</title>`)
    if m := re.FindStringSubmatch(content); len(m) > 1 {
        return strings.TrimSpace(m[1])
    }
    return ""
}
```

For JSX files, we look for the `export default function NAME` pattern:
```go
func extractJSXComponentName(path string) string {
    f, err := os.Open(path)
    if err != nil {
        return ""
    }
    defer f.Close()
    
    scanner := bufio.NewScanner(f)
    re := regexp.MustCompile(`export\s+default\s+function\s+(\w+)`)
    for scanner.Scan() {
        if m := re.FindStringSubmatch(scanner.Text()); len(m) > 1 {
            return m[1]
        }
    }
    return ""
}
```

### Component 2: HTTP Handlers

**File:** `pkg/server/handlers.go`

The server has three route groups:

#### Route: `GET /` — Index Page

Returns an HTML page listing all artifacts. The server re-scans the directory on each request (cheap for small directories, ensures dynamic discovery).

```go
func (s *Server) handleIndex(w http.ResponseWriter, r *http.Request) {
    artifacts, err := s.scanner.Scan()
    if err != nil {
        http.Error(w, err.Error(), 500)
        return
    }
    
    // Render using a Go html/template
    s.indexTemplate.Execute(w, map[string]interface{}{
        "Artifacts": artifacts,
        "Dir":       s.dir,
    })
}
```

The index page template shows a card grid:


#### Route: `GET /view/:name` — Artifact Viewer

This is the main serving route. Behavior depends on artifact type:

**For HTML artifacts:** Read the file and serve it directly with `Content-Type: text/html`.

```go
func (s *Server) handleView(w http.ResponseWriter, r *http.Request) {
    name := chi.URLParam(r, "name") // or however you extract path params
    
    artifact, err := s.scanner.FindByName(name)
    if err != nil {
        http.NotFound(w, r)
        return
    }
    
    switch artifact.Type {
    case "html":
        // Serve directly
        http.ServeFile(w, r, artifact.Path)
        
    case "jsx":
        // Generate and serve the host page
        jsxSource, err := os.ReadFile(artifact.Path)
        if err != nil {
            http.Error(w, err.Error(), 500)
            return
        }
        s.jsxHostTemplate.Execute(w, map[string]interface{}{
            "Title":     artifact.Title,
            "Name":      artifact.Name,
            "JSXSource": string(jsxSource),
        })
    }
}
```

**For JSX artifacts:** Generate the host HTML page described in the architecture section above.

The JSX host template (embedded in Go):

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{{.Title}} — Artifact Viewer</title>
  <script type="importmap">
  {
    "imports": {
      "react": "https://esm.sh/react@18.3.1",
      "react-dom": "https://esm.sh/react-dom@18.3.1",
      "react-dom/client": "https://esm.sh/react-dom@18.3.1/client",
      "react/jsx-runtime": "https://esm.sh/react@18.3.1/jsx-runtime"
    }
  }
  </script>
  <script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
  <style>
    /* Reset to give artifacts a clean slate */
    * { margin: 0; padding: 0; box-sizing: border-box; }
    html, body, #root { width: 100%; height: 100%; }
  </style>
</head>
<body>
  <div id="root"></div>
  <script type="text/babel" data-type="module">
{{.JSXSource}}

// Auto-mount the default export
import { createRoot } from "react-dom/client";
const root = createRoot(document.getElementById("root"));
root.render(<App />);
  </script>
</body>
</html>
```

**Important detail about JSX template injection:** The JSX source is injected directly into the HTML template. This means we need to be careful about HTML entities in the JSX source. In practice, because the source is inside a `<script>` tag, the only character sequence that would cause problems is `</script>`. We should escape that:

```go
// Escape </script> in JSX source to prevent premature tag closing
safeSource := strings.ReplaceAll(jsxSource, "</script>", "<\\/script>")
```

#### Route: `GET /raw/:name` — Raw File Serving

Serves the artifact file as-is with appropriate MIME type. Useful for debugging or downloading.

```go
func (s *Server) handleRaw(w http.ResponseWriter, r *http.Request) {
    name := chi.URLParam(r, "name")

    if err != nil {
        http.NotFound(w, r)
        return

}
```

### Component 3: Server Struct

**File:** `pkg/server/server.go`

```go
type Server struct {
    scanner         *artifacts.Scanner
    dir             string
    port            int
    indexTemplate   *template.Template
    jsxHostTemplate *template.Template
}

type Config struct {
    Dir   string
    Port  int
    Watch bool
}

func New(cfg Config) (*Server, error) {
    scanner := artifacts.NewScanner(cfg.Dir)
    
    // Parse embedded templates
    indexTmpl, err := template.New("index").Funcs(templateFuncs).Parse(indexHTML)
    if err != nil {
        return nil, err
    }
    jsxHostTmpl, err := template.New("jsx-host").Parse(jsxHostHTML)
    if err != nil {
        return nil, err
    }
    
    return &Server{
        scanner:         scanner,
        dir:             cfg.Dir,
        port:            cfg.Port,
        indexTemplate:   indexTmpl,
        jsxHostTemplate: jsxHostTmpl,
    }, nil
}

func (s *Server) Run(ctx context.Context) error {
    mux := http.NewServeMux()
    mux.HandleFunc("GET /", s.handleIndex)
    mux.HandleFunc("GET /view/{name}", s.handleView)
    mux.HandleFunc("GET /raw/{name}", s.handleRaw)
    
    srv := &http.Server{
        Addr:    fmt.Sprintf(":%d", s.port),
        Handler: mux,
    }
    
    // Graceful shutdown on context cancellation
    go func() {
        <-ctx.Done()
        shutdownCtx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
        defer cancel()
        srv.Shutdown(shutdownCtx)
    }()
    
    log.Printf("Serving artifacts from %s on http://localhost:%d", s.dir, s.port)
    return srv.ListenAndServe()
}
```

#### The `serve` Command

This is a plain Cobra command (not a Glazed command) because it runs a long-lived server rather than producing structured output.

**File:** `cmd/serve-artifacts/cmds/serve.go`

```go
func newServeCmd() *cobra.Command {
    var port int
    var dir string
    var watch bool
    
    cmd := &cobra.Command{
        Use:   "serve",
        Short: "Start the artifact server",
        Long: `Start an HTTP server that serves Claude.ai artifacts.

HTML artifacts are served directly. JSX artifacts are wrapped in a host
page that provides React and Babel for in-browser transformation.

Examples:
  serve-artifacts serve --dir ./imports --port 8080
  serve-artifacts serve --dir ~/claude-artifacts --watch`,
        RunE: func(cmd *cobra.Command, args []string) error {
            cfg := server.Config{
                Dir:   dir,
                Port:  port,
                Watch: watch,
            }
            srv, err := server.New(cfg)
            if err != nil {
                return err
            }
            return srv.Run(cmd.Context())
        },
    }
    
    cmd.Flags().IntVarP(&port, "port", "p", 8080, "Port to listen on")
    cmd.Flags().StringVarP(&dir, "dir", "d", ".", "Directory containing artifacts")
    cmd.Flags().BoolVarP(&watch, "watch", "w", false, "Watch for file changes and notify browser")
    
    return cmd
}
```

#### The `list` Command (Glazed)

This uses the full Glazed command framework to output artifact metadata as structured data (table, JSON, YAML, CSV, etc.).

**File:** `cmd/serve-artifacts/cmds/list.go`

```go
type ListCommand struct {
    *cmds.CommandDescription
}

type ListSettings struct {
    Dir  string `glazed:"dir"`
    Type string `glazed:"type"`
}

func NewListCommand() (*ListCommand, error) {
    glazedSection, err := settings.NewGlazedSchema()
    if err != nil {
        return nil, err
    }
    
    return &ListCommand{
        CommandDescription: cmds.NewCommandDescription(
            "list",
            cmds.WithShort("List all artifacts in a directory"),
            cmds.WithFlags(
                fields.New("dir", fields.TypeString,
                    fields.WithDefault("."),
                    fields.WithHelp("Directory containing artifacts"),
                ),
                fields.New("type", fields.TypeString,
                    fields.WithDefault(""),
                    fields.WithHelp("Filter by type (html, jsx)"),
                ),
            ),
            cmds.WithSections(glazedSection),
        ),
    }, nil
}

func (c *ListCommand) RunIntoGlazeProcessor(
    ctx context.Context,
    vals *values.Values,
    gp middlewares.Processor,
) error {
    s := &ListSettings{}
    if err := vals.DecodeSectionInto(schema.DefaultSlug, s); err != nil {
        return err
    }
    
    scanner := artifacts.NewScanner(s.Dir)
    artifacts, err := scanner.Scan()
    if err != nil {
        return err
    }
    
    for _, a := range artifacts {
        if s.Type != "" && a.Type != s.Type {
            continue
        }
        row := types.NewRow(
            types.MRP("name", a.Name),
            types.MRP("filename", a.Filename),
            types.MRP("type", a.Type),
            types.MRP("title", a.Title),
            types.MRP("size", a.Size),
            types.MRP("modified", a.ModifiedAt),
        )
        if err := gp.AddRow(ctx, row); err != nil {
            return err
        }
    }
    return nil
}
```

### Template Embedding

Templates are embedded into the Go binary using `go:embed`:

**File:** `pkg/server/templates.go`

```go
package server

import "embed"

//go:embed templates/index.html
var indexHTML string

//go:embed templates/jsx-host.html
var jsxHostHTML string
```

Alternatively, if templates are in a separate top-level `templates/` directory, you can embed them from the `cmd/` package and pass them to the server constructor. The key point is: templates are compiled into the binary, so the resulting tool is a single executable with no external file dependencies.

## Risks, Alternatives, and Open Questions

### Risks

1. **CDN dependency for JSX**: The JSX host page loads Babel and React from CDNs. If the user is offline, JSX artifacts won't render. Mitigation: we could bundle Babel and React into the Go binary (they're ~2MB compressed) as a `go:embed` fallback.

2. **Babel standalone performance**: In-browser Babel transformation adds ~200-500ms to first load. Mitigation: acceptable for dev tool; add server-side compilation as Phase 5 if needed.

3. **JSX artifact compatibility**: Some artifacts might use React features or imports not covered by our import map (e.g., `react-router`, `framer-motion`). Mitigation: extend the import map as needed; log clear error messages when imports fail.

### Alternatives Considered

1. **Embed esbuild in Go**: The `esbuild` library has a Go API. We could compile JSX server-side. **Rejected because**: adds significant binary size (~8MB), and the runtime approach is simpler for our use case.

2. **Use Vite dev server**: We could shell out to `npx vite` to serve JSX. **Rejected because**: requires Node.js installation, defeats the purpose of a single Go binary.

3. **Use a SPA framework**: Build a full React frontend for the artifact browser itself. **Rejected because**: over-engineering for a listing page; Go templates are sufficient.

### Open Questions

1. **Should we support `.tsx` files?** TypeScript JSX files would need an additional transformation step (TypeScript → JavaScript). Babel standalone supports TypeScript, so this might work with minimal changes.

2. **Should artifacts be able to reference local assets (images, data files)?** Currently they're assumed self-contained. We could add a convention like `artifacts/business-app/` directories.

3. **Should we add a "view source" feature?** The raw endpoint exists, but a syntax-highlighted source viewer could be useful.
